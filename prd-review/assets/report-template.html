这份文档把功能想得挺清楚——问题、用户故事、优先级都立得住,是个不错的起点。但它还不是 一份能交给开发照着做的 PRD:每个故事只写了"一切顺利时"会怎样,识别结果要写进财务系统却没约定怎么写, 而作为一个 AI 功能,最关键的"识别要多准才算能用、识别错了怎么办"都还没有答案。下面会逐条说清楚,并给出 该怎么补。

审核总结

整体来看,这份文档的产品思路是对的——没有一上来就堆功能,而是先讲清了"财务每月手工录入发票,既慢又容易 出错"这个真实问题,用户故事也写得规范。差距集中在"开发要照着做"还缺的那一层:发票识别在现实里会遇到照片 模糊、字段缺失、金额识别不准,这些文档里一概没说怎么处理;识别结果要写进财务系统,但写哪些字段、什么格式、 出错了怎么办都没约定;最要紧的是,作为一个 AI 功能,它没有定义"识别准到什么程度才算合格",也没安排人工复核—— 这意味着有可能把识别错的金额直接记进账。

至于完成度这个数字:它衡量的是"一份 PRD 该有的东西到位了多少",不是你写了多少工作量——功能部分其实写得 不少。40% 的意思是,上面这几样"开发前必须先定"的东西基本还空着,分主要扣在这里,而不是扣在你已经写好的功能上。 把它们补齐,这份文档很快就能到"可以开工"的程度。

必须补齐(不补就没法开工)

必须补 验收标准只写了"一切顺利"的情况 对应:用户故事

每条验收标准都是"上传 → 识别 → 填入"的顺利路径。但发票识别的麻烦恰恰在意外情况:照片模糊、 多张混在一起拍、某个字段是空的、金额栏识别不出、同一张传了两次。这些不说清楚,上线第一周就会变成线上问题。

怎么补
  • 把每个故事的意外情况过一遍:照片模糊、字段缺失、识别不出、重复上传,各写一条明确的预期行为。
  • 每条都写成"什么情况下 → 系统怎么做"。比如:税额识别不出时,这一栏留空并标"需人工确认",不要猜一个数填进去。
  • 清楚告诉用户:哪些情况系统自动处理,哪些要他来补,而不是默默出错。

必须补 要写进财务系统,却没约定怎么写 对应:技术考量(空缺)

功能要把识别结果填进财务系统,但没说清:写哪些字段、每个字段什么格式(金额几位小数、日期怎么写、 税号多少位)、哪些必填、某个字段空着到底是"本来就没有"还是"没识别出来"。这些不定下来,开发没法动手。

怎么补
  • 列一张字段对照表:要写入的每个字段、类型、格式、是否必填、由谁提供。
  • 给一个真实例子(填好的一条记录长什么样),让识别方和财务系统两边对齐。
  • 特别说清"空值"的含义——是这张发票本就没有这项,还是没识别出来,两者后续处理完全不同。

必须补 没分清"系统出错"和"结果需要人工确认" 对应:技术考量

加密的 PDF、超大或损坏的文件,这种"系统根本处理不了"的情况,和"识别出来了但拿不准、需要人工看 一眼"的情况,处理方式完全不同。文档没区分,结果要么对本来能救的情况直接报错,要么把不靠谱的数据悄悄记进账。

怎么补
  • 分两类分别定行为。第一类"系统处理不了"(加密/损坏/过大):明确报错,并提示用户重新上传。
  • 第二类"处理了但拿不准"(识别把握低、信息不全):结果照常返回,但标"需人工复核",人确认后再入账。
  • 两类各自列出:什么情况触发、系统怎么响应、之后谁来接手。

必须补 作为 AI 功能,没定"多准算合格",也没安排人工兜底 对应:成功指标 / 技术 / 排期

"准确率 95%"听着明确,但没说这个数怎么量、拿什么发票来量,等于一句没法验证的承诺;而且整个流程 全自动、没有人工复核,意味着识别错的金额会直接进账。再者,要量准确率就得先有一批"标准答案"发票和字段对照规则, 这些材料没到位,开发既测不了也开不了工——AI 项目最常卡在这,文档里却没提。

怎么补
  • 先准备一批"标准答案":几十张真实发票,人工标好正确字段,作为衡量识别准不准的尺子。
  • 把"准确率 95%"改成可验证的说法:在这批发票上,关键字段识别准确率达到多少才算过关。
  • 加入人工复核:有把握的自动填,没把握的标出来让人确认;拿不准的字段留空,绝不编一个。
  • 把"标准答案发票、字段对照规则表"当成开工前必须先备好的材料,指定谁负责、什么时候交。

建议完善

建议补 成功指标选错了,而且没有底线 对应:成功指标

这是个帮财务省事的内部工具,"识别准确率"只是过程指标,衡量不了真正的价值;而且光追准确率, 容易靠"多录点数据"刷上去。

怎么补
  • 核心指标换成能体现价值的:比如"识别后无需人工修改就能直接用的比例",或"每周帮财务省下多少小时"。
  • 加一条不能踩的红线:错误入账率不能上升、人工复核的工作量应真的下降,而不是从录入转移到复核。
  • 每个指标先量出当前水平(没有就先统计一段时间),再定目标。

建议补 风险没人负责,工期估算没依据 对应:排期 / 风险

"识别可能不准"只点了一句,没说怎么应对、谁来盯;"1 个月上线"在"标准答案"数据还没备好时, 是个没底气的数字。

怎么补
  • 每条风险补上:发生的可能性和影响、谁负责、真出现了怎么办。
  • 工期给一个区间,并写明前提:比如"标准答案数据到位后,3 到 4 个迭代",而不是一个默认材料都齐了的死日期。

建议补 "快、稳"要落成具体数字 对应:技术考量

"快""稳"没法验收。

怎么补
  • 写成能测的数字,并作为验收标准:单张识别多少秒内完成、单个文件最大多大、多久没响应算超时。

可优化 界面只画了正常情况,缺上线后的验证计划 对应:设计 / 发布

只设计了一切正常时的样子;上线后怎么确认指标达成也没写。

怎么补
  • 给关键页面补上"空白、出错、加载中、数据量极大"几种状态。
  • 写清怎么发(先小范围试用还是全量)、上线后看哪些数据确认省时和准确率达标。

逐节点评

背景与问题 ✅完整

问题讲得清楚,谁受影响、为什么痛都到位了。
  • 可以再补一句量化现状(每月多少张、单张多久、错误率多少),让后面的目标和投入产出有个参照。

成功指标 ⚠️部分

把识别准确率当成了产品成功,缺了能体现价值的指标和底线。
  • 核心指标换成"省了多少人工 / 无需修改即可用的比例";
  • 补一条底线:错误入账率、人工复核工时;
  • 每个指标先有当前数据再定目标。

用户故事 ⚠️部分

故事写得规范、优先级清楚,这是亮点。差距是验收标准只有顺利路径(见上面"必须补"第一条),且有些标准不好核对。
  • 给"自动填入"补上能核对的判据:填入哪些字段、没把握的字段怎么标、用户怎么确认。

技术考量 ❌缺失

这是这份文档最大的空缺,也是最该补的地方:接口怎么传、数据怎么约定、出错怎么分类、性能写成数字,这四样都没有 (见上面"必须补"的第二、三条和"建议补"里的性能项)。建议把这一节重写成"和财务系统怎么对接"的具体约定。

做得好的部分

  • 没有一上来就堆功能,而是先把要解决的问题讲清楚——产品思路的起点是对的。
  • 用户故事从用户视角写,粒度合适,优先级和第一版范围都清楚。
  • 对范围有收敛意识(先做专票,不贪多)。

缺的几乎都集中在"技术怎么落地"和"AI 怎么保证质量"这两块。补上之后, 这份文档就能从一份产品想法,变成开发可以放心照着做的 PRD。

下一步(按优先级)

  1. 补四个必须项:意外情况的验收标准、和财务系统的接口与数据约定、错误分类、AI 的质量标准与人工复核。 建议负责人:产品 + 工程接口人,本周内
  2. 定下"标准答案"数据:确认那批标好字段的发票和对照规则由谁提供、什么时候到位——它决定了能不能开工。 负责人:产品,3 天内拍板
  3. 修正指标:核心指标换成"省时/可用比例",加上底线指标,并开始统计当前水平。 负责人:产品
  4. 补齐建议项和界面各种状态,完成后再评一次——预计可以到"有条件交付"。 负责人:产品
